# 硬件系统与经验总结 (Hardware & Tips)

本文档整理了室内动捕无人机 MPC 控制实验系统搭建过程中的硬件细节、推力补偿原理、可视化调试经验及“踩坑”记录，供后续实验参考。

## 1. 硬件布置与连接

### 地面动捕与网络环境
*   **动作捕捉系统**：房间天花板布置了8个红外动捕摄像头。摄像头发射红外光，射向无人机机身上的回退反射球（马克点），产生高对比度反射，通过计算机视觉算法解算出无人机在空间中的六自由度位置和姿态。
*   **有线数据传输**：由于动捕图像数据量巨大（带宽需求高），摄像头通过有线路径连接到交换机，再送至地面电脑进行数据解算。
*   **无线连接（通信中枢）**：地面电脑解算的位姿结果通过有线连接送至路由器，路由器通过 Wi-Fi（无线方式）将数据下发给飞行中的机载电脑。同时，路由器也支持实验人员通过远程桌面（Nomachine）或 SSH 对无人机进行无线调试。

### 无人机机上系统
*   **计算核心（上位机）**：采用计算能力较强的 Jetson NX，负责运行 ROS 系统、状态过滤、MPC 求解及轨迹规划。
*   **飞行控制（下位机）**：采用基于 STM32 的 Pixhawk (PX4) 飞控，负责基础的姿态控制和电机驱动。
*   **动力系统**：包括电子调速器（ESC）、无刷电机和螺旋桨。
*   **通信接口**：Jetson NX 与 PX4 通过串口进行 MAVROS 通信。

## 2. 推力映射在线估计 (Thrust Estimator) 与延迟补偿
实验中解决的一个关键问题是电池电压下降导致的静态误差增大和飞行漂移。
*   **问题**：随着电池电压下降，相同的油门输出对应的实际推力会减小。
*   **方案**：利用带有遗忘因子的递归最小二乘法 (RLS)，在线实时估计“油门值—推力”的映射系数。
*   **延迟补偿**：指令下发到电机响应，再到 IMU 测量出加速度，存在物理和通信上的时间延迟（通常在 30~50ms 之间）。因此，每次下发油门指令时记录时间戳和油门量，仅提取距今 35ms~45ms 之间的历史油门数据与当前的 Z 轴加速度进行匹配，从而保证模型估计的准确性。

## 3. MPC 调试与可视化数据分析
在实飞或播放 rosbag 时，可以通过 RViz 和 PlotJuggler 排查无人机悬停不稳、轨迹跟踪发散等问题。

### 空间域诊断 (RViz)
*   **操作**：订阅 `/odom`（真实位姿）、`/mpc_debug/acado_ref_path`（期望轨迹）、`/mpc_debug/acado_pred_path`（预测轨迹）。
*   **现象分析**：
    *   **正常悬停**：期望轨迹和预测轨迹重合，且紧密包围在 `/odom` 周围。
    *   **预测发散（弹簧效应）**：预测轨迹像弹簧一样疯狂甩动。通常是因为惩罚权重（如 `cost_vx`, `cost_wx`）设置不合理，或预测步长过大而控制频率过低。
    *   **稳态误差**：预测轨迹稳定但偏离 `/odom`。通常是因为悬停推力估计不准，或存在外部风扰/重心偏移。

### 时间域诊断 (PlotJuggler)
*   **位置跟踪误差**：对比 `/odom` 和 `/mpc_debug/ref_pose` 的曲线。如果 `odom` 出现剧烈毛刺，说明 VRPN 数据丢包或动捕刚体遮挡。
*   **控制量饱和分析**：观察 `/mpc_debug/raw_control/data`。如果推力或角速度经常顶到你设置的上限或下限，MPC 将失去调节能力，飞机极易失控。
*   **状态机异常跳变**：监控 `/mpc_debug/mode/data`，观察失控瞬间是否发生状态机的意外打断（例如 Hover 意外掉到 Takeoff）。

## 4. 经验总结（踩坑记录）
1.  **代码选型**：最初尝试使用 UZH-RPG 的代码，但因其结构过于工程化（复杂）且硬件匹配度低而放弃，转而自建轻量化 ROS1 系统。
2.  **飞控设置**：必须深入学习 PX4 的参数设置，特别是如何正确切换到并维持在 OFFBOARD 模式。
3.  **单位统一**：动捕系统与 PX4 的数据发布单位（如米、厘米、弧度、角度）必须严格匹配。
4.  **状态估计**：MPC 对速度估计的准确性非常敏感，必须保证滤波器的性能。
5.  **时变补偿**：必须重视推力映射系数随时间（电压）变化的问题，否则难以实现长时稳定的轨迹跟踪。
